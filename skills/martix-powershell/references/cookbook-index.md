# PowerShell cmdlet cookbook index

## Purpose

Secondary recipe index linking advanced integration scenarios, testing
patterns, help authoring, packaging, and CI/CD patterns for compiled PowerShell
cmdlet development. These are referenced scenarios that extend beyond the core
rule domains. Each topic below indicates the relevant core maps and notes
whether a dedicated rule file exists.

---

## Testing

### Cmdlet unit testing with Pester

Test compiled cmdlets by importing the module into a Pester test file and using
`InModuleScope` to access internal state. Common patterns:

- Import the compiled `.psd1` manifest with `Import-Module` and call cmdlets
  directly in `BeforeAll` / `It` blocks.
- Use `Should -Invoke` (Pester v5+) to assert `WriteObject`, `WriteError`, and
  `ThrowTerminatingError` calls by mocking the underlying service layer.
- Use `Should -Throw` to verify that `ThrowTerminatingError` raises a
  terminating error with the expected `FullyQualifiedErrorId`.
- Test `-WhatIf` behavior by asserting that the mutation service method is
  **not** called when `-WhatIf` is present.

Related maps: [Pipeline map](./pipeline-map.md),
[Error handling and categories map](./error-handling-categories-map.md),
[Confirmation and safety map](./confirmation-safety-map.md)

External resource: [Pester testing framework](https://pester.dev/)

### Cmdlet integration testing with a real PowerShell session

Launch a `PowerShell` subprocess or use `RunspaceFactory` to test the full
module load and pipeline behavior in an isolated session. Integration tests
should cover:

- `Import-Module` succeeds without warnings (no non-approved-verb warnings).
- Parameter binding for all declared parameter sets resolves correctly
  (`Get-Command -Name <Cmdlet> -Syntax`).
- Pipeline pass-through scenarios (`Get-Widget | Set-Widget -PassThru`).
- Error stream output (`$Error`, `$?`) after `WriteError` and
  `ThrowTerminatingError`.

Related maps: [Foundation map](./foundation-map.md),
[Parameters and validation map](./parameters-validation-map.md)

## Comment-based help and `Get-Help`

Add a `<#...#>` XML doc comment block inside the cmdlet class file with at
minimum:

- `.SYNOPSIS` — single-line description.
- `.DESCRIPTION` — detailed explanation.
- `.PARAMETER Name` — one entry per public parameter.
- `.EXAMPLE` — at least one working example with output.
- `.OUTPUTS` — concrete type emitted by `WriteObject`.
- `.NOTES` — version, author, change history.

Validate with `Get-Help <CmdletName> -Full` after loading the module. The
runtime reads help from the XML doc comments embedded in the assembly or from
a companion `<ModuleName>-help.xml` file generated by Platyps.

External resources:
[Platyps help authoring tool](https://github.com/PowerShell/platyPS),
[Comment-based help documentation](https://learn.microsoft.com/en-us/powershell/scripting/developer/help/writing-help-for-windows-powershell-cmdlets)

Dedicated rule for the full documentation, help, and style contract (required
sections, `.OUTPUTS`/`[OutputType]` alignment, alias avoidance, `Read-Host`
and `Write-Host` guidance): [Documentation, help, and style rule](../rules/documentation-help-style.md).

## Credential parameter and `PSCredential` handling

Declare credential parameters as `PSCredential` with the `[Credential]`
attribute:

```csharp
[Parameter]
[Credential]
public PSCredential? Credential { get; set; }
```

The `[Credential]` attribute enables the `[PSCredential]` type transformer,
which allows users to pass a plain username string that PowerShell converts
to a `PSCredential` by prompting for the password interactively. Never accept
a plaintext password `string` parameter.

Related maps: [Parameters and validation map](./parameters-validation-map.md)

External resource:
[CredentialAttribute class](https://learn.microsoft.com/en-us/dotnet/api/system.management.automation.credentialattribute)

## `Path` / `LiteralPath` dual-parameter pattern

Implement both `Path` (wildcard-enabled) and `LiteralPath` (no expansion) as
separate parameter sets:

```csharp
[Parameter(Mandatory = true, ParameterSetName = "Path",
           ValueFromPipelineByPropertyName = true)]
[SupportsWildcards]
public string[] Path { get; set; } = Array.Empty<string>();

[Parameter(Mandatory = true, ParameterSetName = "LiteralPath",
           ValueFromPipelineByPropertyName = true)]
[Alias("PSPath")]
public string[] LiteralPath { get; set; } = Array.Empty<string>();
```

Evaluate wildcards with `WildcardPattern.Get` in the `Path` set; pass
`LiteralPath` values directly without expansion.

Related maps: [Parameters and validation map](./parameters-validation-map.md),
[Pipeline map](./pipeline-map.md)

## Module auto-loading and `ModuleToProcess`

Binary cmdlet modules load via `Import-Module` pointing to the `.psd1`
manifest. PowerShell's module auto-loading discovery (PSModulePath) requires
the manifest to be in a directory named after the module
(`<ModuleName>\<ModuleName>.psd1`). Set `RootModule` in the manifest to the
relative path to the compiled DLL; leave `ModuleToProcess` empty (it is for
script modules).

Related maps: [Foundation map](./foundation-map.md)

## Packaging for the PowerShell Gallery

Before publishing:

1. Ensure `ModuleVersion`, `GUID`, `Author`, `Description`,
   `PowerShellVersion`, and `CmdletsToExport` are populated in the manifest.
2. Run `Test-ModuleManifest -Path .\MyModule.psd1` to validate the manifest.
3. Verify no non-approved-verb warnings appear on `Import-Module -Verbose`.
4. Include a `LICENSE` file and a `README.md` or module description.

External resource:
[Publishing to the PowerShell Gallery](https://learn.microsoft.com/en-us/powershell/gallery/how-to/publishing-packages/publishing-a-package)

## Cross-platform binary module compatibility

Binary modules compiled for `net8.0` run on Windows, Linux, and macOS with
PowerShell 7+. Common compatibility pitfalls:

- Path separators — use `Path.Combine` and `Path.DirectorySeparatorChar`,
  never hardcode `\`.
- Windows-only APIs (`System.Management`, WMI, registry) — guard with
  `RuntimeInformation.IsOSPlatform(OSPlatform.Windows)` and emit a clear
  non-terminating error on unsupported platforms.
- `net8.0` / `net472` multi-targeting — use conditional compilation
  (`#if NET`) to isolate platform-specific code paths.

Related maps: [Foundation map](./foundation-map.md)

## CI/CD pipeline patterns for binary module publishing

Recommended CI/CD flow for a binary cmdlet module:

1. `dotnet restore --nologo` — restore NuGet packages.
2. `dotnet build --nologo -c Release` — compile the assembly.
3. `dotnet test --nologo --no-build` — run unit tests (Pester via
   `dotnet-pester` or Invoke-Pester in a PowerShell step).
4. `dotnet pack -c Release --no-build` — package the module (optional NuGet
   approach) or xcopy the output folder into a publishable layout.
5. `Publish-Module -Path <outputDir> -NuGetApiKey $env:PSGalleryApiKey` —
   publish to the PowerShell Gallery.

Store `PSGalleryApiKey` as a CI secret; never commit it to source control.

External resource:
[PowerShell Gallery publishing](https://learn.microsoft.com/en-us/powershell/gallery/how-to/publishing-packages/publishing-a-package)
