# PowerShell documentation, help, and style

## Purpose

Protect the help, documentation, and script-body style contract for compiled
cmdlets and `[CmdletBinding()]` advanced functions. Accurate help sections
enable `Get-Help` discoverability, `.OUTPUTS`/`[OutputType]` alignment feeds
IDE property inference, script-body alias avoidance keeps automation
consumers safe, and banning `Read-Host` and misused `Write-Host` ensures
functions compose correctly in pipelines and non-interactive environments.

## Default guidance

### Comment-based help blocks (advanced functions)

Place the comment-based help block **immediately inside the function body**,
before the `param` block:

```powershell
function Get-Widget {
    <#
    .SYNOPSIS
    Gets one or more Widget objects by name.

    .DESCRIPTION
    Retrieves Widget objects from the widget store. Accepts pipeline input
    by value and by property name. Emits one Widget per matching name.

    .PARAMETER Name
    One or more widget names to retrieve. Supports wildcards.

    .EXAMPLE
    Get-Widget -Name 'Sprocket'
    Returns the Widget named 'Sprocket'.

    .OUTPUTS
    Widget. One Widget object per matching name.

    .NOTES
    Author: Example Author
    Version: 1.0.0
    #>
    [CmdletBinding()]
    [OutputType([Widget])]
    param(
        [Parameter(Mandatory, ValueFromPipeline, ValueFromPipelineByPropertyName)]
        [ValidateNotNullOrEmpty()]
        [string[]]$Name
    )
    process { ... }
}
```

Required help sections:

| Section | Requirement |
| --- | --- |
| `.SYNOPSIS` | Single-line description; must fit on one line |
| `.DESCRIPTION` | Full explanation; multi-line is encouraged |
| `.PARAMETER <Name>` | One entry per public parameter; describe valid values and defaults |
| `.EXAMPLE` | At least one working command with expected output |
| `.OUTPUTS` | Concrete type name and brief description |
| `.NOTES` | Author, version, and change notes |

`.INPUTS` and `.LINK` are optional but recommended for pipeline-accepting
functions and functions with related online help.

### Help for compiled cmdlets (binary modules)

Binary cmdlet help lives in a companion MAML file
(`<ModuleName>-Help.xml`), typically generated by PlatyPS from Markdown
source. The same required sections apply: Synopsis, Description, Parameters,
Examples, Outputs, Notes. See the
[Cookbook index â€” comment-based help section](../references/cookbook-index.md)
for the PlatyPS generation workflow.

### `.OUTPUTS` and `[OutputType]` alignment

The `.OUTPUTS` text in the help block and the `[OutputType]` attribute must
declare the **same concrete type**. Mismatches break `Get-Help` type display
and IDE property inference on piped results.

| Surface | Attribute declaration | Help section |
| --- | --- | --- |
| Compiled cmdlet (C#) | `[OutputType(typeof(Widget))]` on the class | `.OUTPUTS` in the MAML/PlatyPS source |
| Advanced function (PS) | `[OutputType([Widget])]` in the function header | `.OUTPUTS` in the `<# ... #>` block |

Declaring `[OutputType([object])]` or `[OutputType([PSObject])]` erases IDE
inference. Omitting `[OutputType]` entirely degrades pipeline discoverability
even when the function runs correctly. See
[Attributes and validators â€” OutputType declaration](./attributes-validators-outputtype.md)
for the full attribute guidance.

### Script-body alias avoidance

Inside any function or script body, always use the full cmdlet name and
explicit parameter names. Interactive-session aliases are not acceptable in
`.ps1` or `.psm1` files.

| Alias or short form | Required script-body form |
| --- | --- |
| `?`, `where` | `Where-Object` |
| `%`, `foreach` | `ForEach-Object` |
| `ls`, `dir`, `gci` | `Get-ChildItem` |
| `select` | `Select-Object` |
| `sort` | `Sort-Object` |
| `ft` | `Format-Table` |
| `fl` | `Format-List` |
| `echo` | `Write-Output` |
| `cat` | `Get-Content` |

Use explicit named parameters (`-Name value`) throughout; positional
arguments without a parameter name break readability and are fragile across
PowerShell versions.

### Read-Host avoidance

Never call `Read-Host` inside a function or script body. Automation consumers
(CI/CD pipelines, `Invoke-Command`, scheduled tasks) have no interactive
terminal; `Read-Host` blocks indefinitely. All required input must arrive
through declared parameters. When the calling code needs to prompt a user,
that responsibility belongs at the presentation layer, not inside the
function.

### Write-Host scope

`Write-Host` writes directly to the host console and bypasses all PowerShell
streams. Use it **only** for direct user-interface text â€” interactive banner
messages or REPL decoration that an automation consumer is intentionally not
expected to capture. Never use `Write-Host` to emit:

| What to emit | Correct mechanism |
| --- | --- |
| Pipeline data (objects) | `Write-Output` (script) / `WriteObject` (compiled) |
| Diagnostic trace | `Write-Verbose` / `WriteVerbose` |
| Status information | `Write-Information` / `WriteInformation` |
| Recoverable condition | `Write-Warning` / `WriteWarning` |
| Non-terminating error | `Write-Error` / `WriteError` |
| Terminating error | `$PSCmdlet.ThrowTerminatingError` |

See [Pipeline â€” output and streaming](./pipeline-output-streaming.md) for the
full stream selection guidance.

### Formatting and style pointer (Tier-5 repo overlay)

For generated `.ps1` / `.psm1` code in this repository, apply the
project-specific naming conventions, indentation, brace placement, and
advanced-function structural layout defined in the **Tier-5 style overlay**:
`.github/instructions/powershell.instructions.md`. This overlay governs
script-body style after tier-1 through tier-4 SDK guidance has been applied.
When the overlay conflicts with a tier-1 SDK fact, follow tier 1 and document
the discrepancy. See [Source index and guardrails](../references/doc-source-index.md)

## Avoid

- **Omitting `.OUTPUTS` from the help block or `[OutputType]` from the
  attribute.** Both are required for full discoverability; missing either
  degrades `Get-Help` output and IDE pipeline inference.

- **Declaring `.OUTPUTS` with a type that differs from `[OutputType]`.** The
  two declarations must be consistent; a mismatch misleads IDE tooling and
  `Get-Help -Full` output.

- **Using `Write-Host` to emit pipeline data, status, warnings, or errors.**
  `Write-Host` output cannot be captured or redirected. Any downstream
  `Out-File`, assignment, or `Tee-Object` silently receives nothing.

- **Calling `Read-Host` anywhere in a function or script body.** This
  permanently breaks non-interactive execution; accept all input via
  parameters.

- **Using alias forms in script bodies** (`?`, `%`, `gci`, `ls`, `dir`,
  `select`, `sort`, `ft`, `fl`, `echo`, `cat`). Aliases are for interactive
  shells only.

- **Using positional arguments without explicit parameter names.** Positional
  binding depends on declaration order and breaks when parameters are
  reordered or inherited.

- **Placing the help block outside or adjacent to the function.** For advanced
  functions, the comment-based help block must sit immediately inside the
  function body; placed outside, PowerShell associates it with the wrong
  scope.

## Review checklist

- [ ] Every public advanced function has a `<# ... #>` comment-based help
      block with `.SYNOPSIS`, `.DESCRIPTION`, one `.PARAMETER` entry per
      public parameter, at least one `.EXAMPLE`, `.OUTPUTS`, and `.NOTES`.
- [ ] Compiled cmdlets have a companion MAML help file covering the same
      required sections.
- [ ] `.OUTPUTS` text and `[OutputType(typeof(T))]` (C#) or
      `[OutputType([T])]` (PS) declare the same concrete type; neither uses
      `[object]` or `[PSObject]`.
- [ ] No `Read-Host` calls appear anywhere in the function or script body.
- [ ] `Write-Host` appears only in code paths that produce direct
      user-interface text; all pipeline data, diagnostics, warnings, and
      errors route through the appropriate stream.
- [ ] All script-body cmdlet references use full names (`Where-Object`,
      `ForEach-Object`, `Get-ChildItem`, etc.); no alias forms are present.
- [ ] All parameter invocations in the script body use explicit named form
      (`-Parameter value`); no positional argument shorthand.
- [ ] Generated `.ps1` / `.psm1` code conforms to the Tier-5 style overlay
      at `.github/instructions/powershell.instructions.md`.
