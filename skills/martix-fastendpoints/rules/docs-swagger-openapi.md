# Swagger and OpenAPI with FastEndpoints

## Purpose

Guide FastEndpoints API documentation toward the built-in Swagger/NSwag
integration, accurate endpoint metadata, and reviewable docs configuration
instead of ad-hoc OpenAPI wiring.

## Default guidance

- Install and use `FastEndpoints.Swagger`, configure documents through the
  FastEndpoints extensions, and keep `UseSwaggerGen()` after
  `UseFastEndpoints()` in the pipeline.
- Start with FastEndpoints' inferred `Accepts` and `Produces` metadata, then
  override only the parts that are wrong by using `Description(...)`,
  `Accepts(...)`, `Produces(...)`, or the clear-default helpers.
- Put endpoint summaries, request examples, response examples, and parameter
  descriptions in `Summary(...)`, `EndpointSummary`, or `Summary<TEndpoint>`
  types when that keeps handlers focused on transport logic.
- Use FastEndpoints-specific documentation attributes and hooks such as
  `[QueryParam]`, `[DefaultValue]`, `[HideFromDocs]`, auth scheme configuration,
  tags, and endpoint naming rather than hand-editing generated OpenAPI output.
- Treat client generation and `swagger.json` export as startup concerns: place
  the generation/export calls immediately after `UseFastEndpoints()` so the app
  can exit cleanly in generation mode before other expensive startup work runs.

## Avoid

- Do not call NSwag's plain `.AddSwaggerDocument()` extension instead of the
  FastEndpoints swagger registration path.
- Do not put `UseSwaggerGen()` before `UseFastEndpoints()` or leave docs
  middleware ordering implicit.
- Do not clear all default metadata unless there is a concrete mismatch such as
  incorrect content types or unwanted autogenerated responses.
- Do not assume System.Text.Json custom converters flow into NSwag schema or
  example generation automatically; review converter or type-mapper support when
  docs depend on them.

## Review checklist

- Swagger registration, middleware order, and document naming are explicit and
  use the FastEndpoints integration points.
- `Accepts`, `Produces`, auth, tags, and operation IDs match the real endpoint
  transport contract.
- Summaries and examples live in a reusable location when endpoint classes would
  otherwise become documentation-heavy.
- Any query-bound or hidden DTO properties use the FastEndpoints attributes that
  keep the generated schema honest.
- Client generation or swagger export steps are placed early enough in startup
  to avoid side effects in generation mode.
